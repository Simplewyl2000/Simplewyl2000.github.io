DEEPBINDIFF: Learning Program-Wide Code Representations for Binary Diffifing

Abstract

Binary diffing analysis quantitatively measures the differences between two given binaries and produces fine-grained basic block level matching. It has been widely used to enable different kinds of critical security analysis. However, all existing program analysis and machine learning based techniques suffer from low accuracy, poor scalability, coarse granularity, or require extensive labeled training data to function. In this paper, we propose an unsupervised program-wide code representation learning technique to solve the problem. We rely on both the code semantic information and the program-wide control flow information to generate basic block embeddings. Furthermore, we propose a k hop greedy matching algorithm to find the optimal diffing results using the generated block embeddings. We implement a prototype called DEEPBINDIFF and evaluate its effectiveness and efficiency with a large number of binaries. The results show that our tool outperforms the state-of-the-art binary diffing tools by a large margin for both cross-version and cross-optimization-level diffing. A case study for OpenSSL using real-world vulnerabilities further demonstrates the usefulness of our system.

Key problem

Existing method including traditional approaches(static analysis, dynamic analysis) and learning-based approaches both have drawbacks, such as low coverage, low efficiency(Traditional approaches), poor dataset and sensitive to the compile settings(learning-based), and can not handle the information from both program-wide perspective and BB perspective(both)

Input

  • Given binary pieces

Output

  • Similarity score between given binary pieces

Detailed Technic

Major tech can be divided into two parts, one is the representation of the BB, the other is the comparison between the BB

And these two parts focus on two major aspects.

  • Node representation: How to deal with the relationship among BBs while taking the instructions in BBs into consideration.
  • Code diffing: Overcome the drawbacks of linear assignment) low efficiency. 2) no graph information considered. 3) Affected by the compile settings.

Pre-process

  • Extract ICFG: Using IDA pro

  • Generate the feature vector:

    • Random walk: To get the context information for the instructions, and has its method to make sure that the random walk has enough completeness

    • Normalize: Replace the specific numbers and variables to certain names.

    • Model Training

    • Calculate feature vector:

      • Concatenate the opcode withe the operands to the vector of an instruction
      • sum all the vectors in one BB as the feature vector of the BB
      • Using the weight parameter to alleviate the influence caused by the compile settings(like O1, O3 optimization)

Node representation

Graph Merging

  • why merge

    • When taking two graph together into consideration, more detailed similarity can be found
    • Merge two into one can boost the efficiency

  • How to merge:

    • Create virtual nodes for library function call or string reference

    • Nodes from two graphs who call the same function point to the same virtual node.

BB embeddings

M: where each entry Mij is logarithm of the average probability that vertex vi randomly walks to vertex vj in fixed steps.

W: The embedding for the nodes from the perspective of the graph

T: Text feature vector of the BB

W concatenate with HT to be a 2k dimension vector, which is the final vector for the BB embedding.

On that account, each generated basic block embedding, contains not only the semantic information about the basic block itself, but also the information from the ICFG structure

Code Diffing

k-Hop Greedy Matching

ComputeInitialSet: extract direct neighbors of the virtual nodes and finds best matching pairs among the neighbors based on embeddings.

GetKHopNeighbors(): exploring the merged ICFG. It then sorts the similarities between neighbor basic blocks.

FindMaxUnmatched(): picks the pair that bears highest similarity with a predefined threshold t of 0.6

As for the dead node or low similarity score node, linear assignment using Hungarian algorithm will be performed.
